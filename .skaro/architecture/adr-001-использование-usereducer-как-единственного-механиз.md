# ADR-001: Использование useReducer как единственного механизма управления игровым состоянием

**Status:** accepted
**Date:** 2026-02-28

## Context
Игровое состояние включает несколько взаимосвязанных полей: текущий экран (`AppScreen`), состояние доски (`BoardState`), статус игры (`GameStatus`) и настройки (`GameSettings`). Все эти поля должны обновляться атомарно — например, при ходе игрока одновременно меняются доска, статус и, возможно, экран. Использование отдельных `useState` для каждого поля создаёт риск рассинхронизации между рендерами и затрудняет тестирование логики переходов.

## Decision
Всё игровое состояние хранится в единственном объекте `GameState` и изменяется исключительно через `dispatch(action: GameAction)` в компоненте `App`. Редьюсер `gameReducer` является чистой функцией `(state, action) => state`, что позволяет тестировать все переходы состояния изолированно от React. Прямая мутация состояния запрещена как архитектурный инвариант.

## Alternatives
1. **Несколько useState для каждого поля состояния:** простой подход для небольших форм — отклонён, поскольку атомарное обновление нескольких полей (доска + статус + экран) невозможно гарантировать без дополнительных ухищрений; высок риск рассинхронизации между рендерами.
2. **Внешний стор (Zustand, Redux Toolkit):** предоставляет мощные инструменты DevTools и middleware — отклонён, поскольку добавляет внешнюю зависимость и избыточную сложность для однопользовательского приложения с мелким деревом компонентов; `useReducer` покрывает все потребности без дополнительных пакетов.
3. **React Context + useState:** позволяет избежать prop drilling — отклонён, поскольку дерево компонентов достаточно мелкое и не требует Context; кроме того, Context не решает проблему атомарности обновлений.

## Consequences
- Positive: атомарность обновлений гарантирована; редьюсер легко тестируется как чистая функция; единственная точка мутации упрощает отладку и понимание потока данных; все переходы состояния явно описаны через дискриминированный union `GameAction`.
- Negative: несколько больше шаблонного кода по сравнению с `useState`; для добавления нового действия требуется изменить тип `GameAction`, редьюсер и тесты.
- Risks: при значительном росте числа действий редьюсер может стать трудночитаемым — в этом случае потребуется разбивка на вложенные редьюсеры или переход на внешний стор (с созданием нового ADR).
