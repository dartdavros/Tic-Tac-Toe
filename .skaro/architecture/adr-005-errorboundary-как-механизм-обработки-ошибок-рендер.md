# ADR-005: ErrorBoundary как механизм обработки ошибок рендеринга

**Status:** accepted
**Date:** 2026-02-28

## Context
В React 18 ошибки, возникающие во время рендеринга дочерних компонентов, не перехватываются стандартными конструкциями `try/catch` в функциональных компонентах. Без явного механизма обработки такие ошибки приводят к «белому экрану» и полной потере интерактивности. Необходимо обеспечить graceful degradation: показать понятное сообщение об ошибке и дать пользователю возможность восстановить работу приложения.

## Decision
Компонент `ErrorBoundary` (классовый компонент, единственный допустимый способ перехвата ошибок рендеринга в React) оборачивает всё дерево ниже `App`. При исключении отображается `ErrorFallback` с кнопкой «Начать заново», которая вызывает `dispatch({ type: 'RESTART' })` и сбрасывает границу ошибки. Ошибка логируется через `logEvent('render_error', error)`. Ошибки в обработчиках событий и синхронном коде редьюсера обрабатываются отдельно — защитным `try/catch` внутри `gameReducer`.

## Alternatives
1. **Отсутствие ErrorBoundary (только try/catch в обработчиках):** минимальный код — отклонён, поскольку `try/catch` не перехватывает ошибки рендеринга React; при исключении в компоненте пользователь видит пустой экран без возможности восстановления.
2. **Несколько ErrorBoundary на уровне каждого экрана:** более гранулярное восстановление — отклонён как избыточный для приложения с тремя экранами; единственный `ErrorBoundary` на уровне корня достаточен и проще в поддержке.
3. **Сторонняя библиотека (react-error-boundary):** удобный API, поддержка хуков — отклонён, поскольку добавляет зависимость; собственная реализация тривиальна для данного случая и не требует дополнительного пакета.

## Consequences
- Positive: пользователь никогда не видит «белый экран смерти»; кнопка восстановления позволяет продолжить работу без перезагрузки страницы; ошибки логируются для последующего анализа.
- Negative: `ErrorBoundary` требует классового компонента, что нарушает единообразие кодовой базы (все остальные компоненты функциональные); не перехватывает ошибки в обработчиках событий и асинхронном коде.
- Risks: если ошибка возникает в самом `ErrorFallback`, приложение всё равно упадёт; необходимо убедиться, что `ErrorFallback` максимально прост и не содержит сложной логики.
