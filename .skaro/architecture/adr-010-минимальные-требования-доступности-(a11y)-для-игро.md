# ADR-010: Минимальные требования доступности (a11y) для игровых элементов

**Status:** accepted
**Date:** 2026-02-28

## Context
Приложение является интерактивной игрой с нестандартными элементами управления (клетки игрового поля). Без явных требований к доступности игра будет недоступна для пользователей скринридеров и клавиатурной навигации. Необходимо определить минимальный набор a11y-требований, реализуемых в рамках учебного проекта без значительных накладных расходов.

## Decision
Определён обязательный минимальный набор a11y-атрибутов для всех интерактивных элементов: клетки `Cell` получают `role="button"`, `aria-label` с описанием позиции и значения, `tabIndex={0}` и обработку `onKeyDown` (Enter / Space); `StatusBar` использует `aria-live="polite"` для объявления изменений статуса; `Board` получает `role="grid"` и `aria-label`; все `<input>` и `<select>` в `GameSettings` имеют явные `<label>`; кнопки в `ResultScreen` используют стандартный `<button>` с понятным текстовым содержимым. Наличие `role="button"` и `aria-label` в `Cell` проверяется smoke-тестами.

## Alternatives
1. **Полное соответствие WCAG 2.1 AA:** максимальная доступность — отклонён как избыточный для учебного проекта; требует значительных дополнительных усилий (управление фокусом, объявления о ходах ИИ, полная клавиатурная навигация по сетке с arrow keys); выходит за рамки учебных целей.
2. **Отсутствие явных a11y-требований:** минимальный код — отклонён, поскольку нестандартные элементы (div/span вместо button) полностью недоступны для скринридеров и клавиатурных пользователей; даже минимальные атрибуты существенно улучшают доступность.
3. **Использование нативных HTML-элементов везде (только `<button>` для клеток):** максимальная семантика без дополнительных ARIA-атрибутов — частично принят (кнопки в `ResultScreen` используют стандартный `<button>`); для клеток поля `role="button"` на `<div>` выбран для большей гибкости стилизации, хотя нативный `<button>` был бы предпочтительнее с точки зрения a11y.

## Consequences
- Positive: игра доступна для пользователей скринридеров и клавиатурной навигации на базовом уровне; `aria-live` в `StatusBar` обеспечивает объявление смены хода и результата игры; a11y-атрибуты проверяются автоматическими тестами.
- Negative: реализован только минимальный уровень доступности; навигация по сетке с помощью клавиш-стрелок (стандарт для `role="grid"`) не реализована — пользователи скринридеров вынуждены использовать Tab для перемещения между клетками.
- Risks: при аудите доступности приложение не пройдёт полную проверку WCAG 2.1 AA; если целевая аудитория включает пользователей с ограниченными возможностями, потребуется расширение a11y-требований и создание нового ADR.
