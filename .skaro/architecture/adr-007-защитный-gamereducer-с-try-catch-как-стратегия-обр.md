# ADR-007: Защитный gameReducer с try/catch как стратегия обработки ошибок в бизнес-логике

**Status:** accepted
**Date:** 2026-02-28

## Context
`ErrorBoundary` перехватывает только ошибки рендеринга React-компонентов. Ошибки в синхронном коде обработчиков событий и в редьюсере (включая вызов `getBestMove`) не перехватываются `ErrorBoundary` и при отсутствии защиты приводят к неконтролируемому краху приложения. Необходима стратегия, обеспечивающая стабильность состояния при непредвиденных ошибках в логике.

## Decision
`gameReducer` оборачивает всю логику обработки действий в `try/catch`. При перехвате исключения редьюсер возвращает текущее состояние без изменений и логирует ошибку через `logEvent('reducer_error', { action, error })`. Это гарантирует, что состояние приложения никогда не окажется в неконсистентном виде из-за непредвиденного исключения. Дополнительно все чистые функции в `logic/` обязаны явно обрабатывать граничные случаи без выброса исключений там, где это возможно.

## Alternatives
1. **Отсутствие try/catch в редьюсере (исключения всплывают):** минимальный код — отклонён, поскольку исключение в редьюсере не перехватывается `ErrorBoundary` и приводит к неконтролируемому краху; состояние может остаться в частично обновлённом виде.
2. **Валидация всех входных данных до вызова редьюсера (в компонентах):** предотвращение ошибок на входе — отклонён, поскольку нарушает инвариант «компоненты не выполняют валидацию самостоятельно»; дублирует логику валидации между компонентами и редьюсером; не защищает от программных ошибок внутри самого редьюсера.
3. **Глобальный обработчик window.onerror / unhandledrejection:** перехват на уровне браузера — отклонён как недостаточный, поскольку не позволяет восстановить состояние приложения; подходит только для логирования, но не для graceful recovery.

## Consequences
- Positive: состояние приложения никогда не становится неконсистентным из-за непредвиденного исключения; пользователь может продолжить игру даже при программной ошибке; все ошибки логируются для последующего анализа.
- Negative: «проглатывание» исключений может затруднить отладку в development-режиме; разработчик может не заметить ошибку, если не следит за консолью; редьюсер технически перестаёт быть «чистой» функцией из-за побочного эффекта логирования.
- Risks: если исключение возникает систематически (например, из-за бага в `getBestMove`), редьюсер будет молча возвращать старое состояние при каждом ходе ИИ — игра «зависнет» без видимой причины; необходимо убедиться, что `logEvent` в production направляет ошибки в систему мониторинга.
