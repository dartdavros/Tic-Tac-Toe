# ADR-006: Дискриминированный union для GameSettings вместо опционального поля humanPlayer

**Status:** accepted
**Date:** 2026-02-28

## Context
Приложение поддерживает два режима игры: PvP (два игрока за одним устройством) и PvAI (игрок против компьютера). В режиме PvAI необходимо знать, за какую сторону играет человек (`humanPlayer: Player`). В режиме PvP это поле не имеет смысла. Необходимо выбрать способ типизации настроек, исключающий обращение к `humanPlayer` в PvP-режиме без явной проверки.

## Decision
Тип `GameSettings` определён как дискриминированный union:
```typescript
type GameSettings =
  | { mode: 'pvp' }
  | { mode: 'pvai'; humanPlayer: Player };
```
Поле `humanPlayer` физически отсутствует в ветке `pvp`, что делает обращение к нему без проверки `mode` ошибкой компиляции TypeScript. Это исключает целый класс runtime-ошибок на уровне системы типов.

## Alternatives
1. **Единый интерфейс с опциональным полем (`humanPlayer?: Player`):** проще в определении — отклонён, поскольку TypeScript не запрещает обращение к `settings.humanPlayer` без проверки на `undefined`; при добавлении новых режимов количество опциональных полей растёт, тип становится трудночитаемым.
2. **Единый интерфейс с обязательным полем и значением-заглушкой (`humanPlayer: Player | null`):** явное значение для PvP-режима — отклонён, поскольку `null` семантически означает «нет значения», но не запрещает обращение без проверки; дискриминированный union выражает намерение точнее и безопаснее.

## Consequences
- Positive: невозможно обратиться к `humanPlayer` в PvP-режиме без явной проверки `mode === 'pvai'` — ошибка компиляции; добавление новых режимов (например, `pvai-hard`) требует явного описания их полей; код самодокументируется.
- Negative: при обработке `GameSettings` в редьюсере требуется явная проверка `mode` через `if` или `switch`; незначительно больше кода по сравнению с опциональным полем.
- Risks: при добавлении третьего режима игры разработчик должен явно добавить новую ветку union и обновить все `switch`-выражения — компилятор предупредит о неполном покрытии только при использовании exhaustive checks (например, через вспомогательную функцию `assertNever`).
