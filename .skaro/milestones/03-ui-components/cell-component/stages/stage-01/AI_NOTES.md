## Почему такой подход

### Нативный `<button>` вместо `<div role="button">`
Согласно уточнению Q1 и FR-03, используется нативный `<button>`. Это даёт:
- Бесплатную поддержку Enter/Space без `onKeyDown`
- Нативный атрибут `disabled`, блокирующий клики на уровне браузера
- Корректный `role="button"` по умолчанию без явного указания

### Двойная защита от клика при `disabled`
Несмотря на то что нативный `<button disabled>` блокирует клики в браузере,
в jsdom (среда Vitest) поведение может отличаться. Поэтому в `handleClick`
добавлена явная проверка `if (!disabled)`. Это соответствует FR-06 и
гарантирует корректное поведение в тестах.

### `aria-disabled` всегда присутствует
Согласно уточнению Q2 и FR-10, `aria-disabled={disabled}` передаётся всегда
(со значением `false` по умолчанию). Это явно отражает состояние доступности
для вспомогательных технологий независимо от нативного атрибута `disabled`.

### CSS-классы через условный рендеринг
Согласно уточнению Q5 и FR-11, используется схема: базовый класс `cell` +
один из `cellX` / `cellO` / `cellEmpty`. Это позволяет легко добавить
подсветку победной линии в будущем через дополнительный класс в `Board`.

### Стили только через CSS-переменные
Все цвета, размеры и шрифты берутся из CSS-переменных `:root` (инвариант №15).
Хардкод значений в `Cell.module.css` отсутствует.

### Тесты CSS-классов с identity-obj-proxy
В тестовой среде Vitest CSS Modules обрабатываются через `identity-obj-proxy`,
который возвращает имя класса как строку (например, `styles.cellX` → `"cellX"`).
Тесты проверяют наличие/отсутствие классов через `className.split(/\s+/)`,
что корректно работает с этим прокси.

## Риски и ограничения

### Поведение `disabled` в jsdom
Нативный `<button disabled>` в jsdom блокирует клики корректно при использовании
`userEvent` из `@testing-library/user-event` (версия 14+). Если тесты на
`disabled` падают — нужно проверить версию `@testing-library/user-event`.
Добавлена явная проверка `if (!disabled)` в `handleClick` как дополнительная
страховка.

### CSS Modules в тестах
Если `identity-obj-proxy` не настроен в `vitest.config.ts`, CSS-классы
будут возвращать `undefined`. В этом случае тесты на классы упадут.
Предполагается, что `src/test-setup.ts` и конфиг Vitest уже настроены
корректно (согласно описанию существующих тестов `Board.test.tsx`).

### Совместимость с `Board.tsx`
Существующий `Board.tsx` передаёт пропс `onClick` (не `onCellClick`) в `Cell`.
После перезаписи `Cell.tsx` интерфейс изменился: теперь используется `onCellClick`.
Это **сломает** `Board.tsx`. Необходимо обновить `Board.tsx` для использования
нового интерфейса `CellProps`. Однако согласно правилам задания, в Stage 1
реализуется только компонент Cell — исправление Board входит в другой этап.

**Альтернатива:** оставить пропс с именем `onClick` для совместимости с Board.
Но спецификация FR-01 явно требует `onCellClick: (index: number) => void`,
поэтому выбрано точное следование спецификации.
