# Clarifications: cell-component

## Question 1
Использовать нативный <button> или <div role='button'> в качестве корневого элемента?

*Context:* Нативный <button> автоматически обрабатывает Enter/Space и доступность без дополнительного onKeyDown, что упрощает реализацию и улучшает a11y; выбор влияет на структуру компонента и объём кода.

**Options:**
- A) Использовать нативный <button> — убрать onKeyDown, получить нативную поддержку клавиатуры бесплатно
- B) Использовать <div role='button'> с явным onKeyDown — строго следовать FR-03 как написано в спецификации
- C) Использовать <button>, но сохранить onKeyDown для явности и тестируемости

**Answer:**
Использовать нативный <button> — убрать onKeyDown, получить нативную поддержку клавиатуры бесплатно

## Question 2
Нужно ли добавлять aria-disabled={true} при disabled={true}?

*Context:* Без aria-disabled скринридер не сообщает пользователю, что клетка недоступна; это влияет на a11y-соответствие ADR-010 и на acceptance criteria.

**Options:**
- A) Да — добавить aria-disabled={disabled} всегда, когда disabled передан
- B) Да — добавить aria-disabled только когда disabled === true, иначе не выставлять атрибут
- C) Нет — достаточно tabIndex={-1} и отсутствия реакции на клик

**Answer:**
Да — добавить aria-disabled={disabled} всегда, когда disabled передан

## Question 3
Что должен содержать aria-label для пустой клетки и клетки с X/O — конкретный формат строки?

*Context:* FR-04 задаёт шаблон 'Клетка {row}×{col}, {значение или пусто}', но не уточняет: 'пусто' или 'пустая', 'X' или 'крестик', 'O' или 'нолик' — это влияет на тесты и локализацию.

**Options:**
- A) Использовать точно: 'Клетка 1×1, пусто' / 'Клетка 1×1, X' / 'Клетка 1×1, O'
- B) Использовать человекочитаемые названия: 'Клетка 1×1, пустая' / 'Клетка 1×1, крестик' / 'Клетка 1×1, нолик'
- C) Использовать: 'Клетка 1×1, пусто' / 'Клетка 1×1, крестик' / 'Клетка 1×1, нолик'

**Answer:**
Использовать: 'Клетка 1×1, пусто' / 'Клетка 1×1, крестик' / 'Клетка 1×1, нолик'

## Question 4
Как должен вести себя компонент при клике на disabled-клетку: тихо игнорировать или останавливать всплытие события?

*Context:* FR-06 говорит только 'не вызывает onCellClick', но не уточняет, нужно ли вызывать event.preventDefault() и event.stopPropagation() — это влияет на поведение родительских обработчиков и тесты.

**Options:**
- A) Тихо игнорировать — просто не вызывать onCellClick, событие всплывает дальше
- B) Вызывать event.preventDefault() + event.stopPropagation() чтобы полностью заблокировать событие
- C) Вызывать только event.preventDefault() без остановки всплытия

**Answer:**
Тихо игнорировать — просто не вызывать onCellClick, событие всплывает дальше

## Question 5
Какой CSS-класс применять к клетке в зависимости от значения (X, O, null) — один динамический класс или отдельные классы?

*Context:* FR-07 задаёт цвета через CSS-переменные, но не описывает структуру CSS-классов; выбор влияет на архитектуру Cell.module.css и возможность будущей подсветки победной линии.

**Options:**
- A) Один базовый класс cell + дополнительные классы cellX / cellO / cellEmpty через условный рендеринг
- B) Только базовый класс cell, цвет задаётся через data-атрибут (data-value='X') и CSS-селекторы
- C) Один базовый класс cell + inline style для цвета символа

**Answer:**
Один базовый класс cell + дополнительные классы cellX / cellO / cellEmpty через условный рендеринг

**Options:**
- A) Да — экспортировать интерфейс CellProps из Cell.tsx
- B) Нет — тип используется только внутри компонента, экспорт избыточен

**Answer:**
Да — экспортировать интерфейс CellProps из Cell.tsx
