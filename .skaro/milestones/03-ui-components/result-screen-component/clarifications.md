# Clarifications: result-screen-component

## Question 1
Какой тип GameStatus используется в проекте и как именно выглядит его структура для победы?

*Context:* FR-01 упоминает `status.kind === 'won'` и поле `winner`, но точная форма типа (например, `{ kind: 'won'; winner: Player }` или `{ kind: 'won'; winner: 'X' | 'O' }`) влияет на типизацию компонента и тесты.

**Options:**
- A) GameStatus = { kind: 'won'; winner: 'X' | 'O' } | { kind: 'draw' }
- B) GameStatus = { kind: 'won'; winner: Player } | { kind: 'draw' }, где Player — отдельный тип из общих типов проекта
- C) GameStatus включает также kind: 'idle' и kind: 'playing', компонент получает уже отфильтрованный union только с 'won' | 'draw'

**Answer:**
GameStatus = { kind: 'won'; winner: 'X' | 'O' } | { kind: 'draw' }

## Question 3
Нужно ли добавить атрибут aria-live на контейнер ResultScreen для объявления итога скринридером?

*Context:* ADR-010 требует минимальный набор a11y-атрибутов для интерактивных элементов; Open Questions в спецификации явно поднимают этот вопрос, и его решение влияет на разметку компонента.

**Options:**
- A) Да, добавить aria-live="assertive" на корневой контейнер результата
- B) Да, добавить aria-live="polite" на корневой контейнер результата
- C) Нет, aria-live не нужен — достаточно стандартного фокуса на первой кнопке при появлении экрана
- D) Нет, не нужен — учебный проект, минимальный a11y ограничивается только кнопками

**Answer:**
Да, добавить aria-live="assertive" на корневой контейнер результата
