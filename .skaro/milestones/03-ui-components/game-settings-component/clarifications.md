# Clarifications: game-settings-component

## Question 1
Что происходит с выбранной стороной (X/O) при переключении режима с PvAI обратно на PvP — сбрасывается ли она в значение по умолчанию (X)?

*Context:* Если пользователь выбрал O, затем переключился на PvP, а потом снова на PvAI — неясно, должна ли сторона сохраниться или сброситься, что влияет на логику useState.

**Options:**
- A) Сбрасывать выбор стороны до X при каждом переключении на PvAI
- B) Сохранять последний выбор стороны при возврате к PvAI
- C) Не важно, любое поведение допустимо

**Answer:**
Сбрасывать выбор стороны до X при каждом переключении на PvAI

## Question 2
Как визуально скрывается блок выбора стороны при режиме PvP — через CSS (visibility/display) или через условный рендеринг (не монтируется в DOM)?

*Context:* Спецификация говорит 'скрывается', но не уточняет механизм; это влияет на доступность (a11y) и на то, будут ли скрытые input-ы присутствовать в DOM и мешать обходу по Tab.

**Options:**
- A) Условный рендеринг (блок не монтируется в DOM при PvP)
- B) CSS display:none (блок в DOM, но скрыт)
- C) CSS visibility:hidden (блок в DOM, занимает место, но скрыт)

**Answer:**
Условный рендеринг (блок не монтируется в DOM при PvP)

## Question 3
Должна ли кнопка «Начать игру» быть заблокирована (disabled) в каком-либо состоянии формы?

*Context:* Спецификация явно исключает валидацию формы, но не уточняет, может ли кнопка быть disabled — например, если ни одна радиокнопка не выбрана (хотя по умолчанию PvP выбран всегда).

**Options:**
- A) Кнопка всегда активна (disabled не используется, так как дефолтные значения всегда заданы)
- B) Кнопка disabled до явного взаимодействия пользователя с формой
- C) Не нужно, поведение по умолчанию покрывает все случаи

**Answer:**
Кнопка всегда активна (disabled не используется, так как дефолтные значения всегда заданы)

## Question 4
Какой HTML-элемент использовать для группировки радиокнопок режима и стороны — семантический <fieldset> с <legend> или просто <div> с заголовком?

*Context:* ADR-010 требует минимальный набор a11y-атрибутов; <fieldset>/<legend> является стандартом доступности для групп радиокнопок, но спецификация явно не указывает структуру разметки.

**Options:**
- A) Использовать <fieldset> + <legend> для каждой группы радиокнопок (рекомендуется для a11y)
- B) Использовать <div> + <p>/<span> в качестве заголовка группы (проще, но хуже для скринридеров)
- C) Любой вариант допустим, главное — наличие <label> для каждого <input>

**Answer:**
Использовать <fieldset> + <legend> для каждой группы радиокнопок (рекомендуется для a11y)

## Question 5
Какой минимальный набор smoke-тестов должен быть в GameSettings.test.tsx согласно требованиям проекта?

*Context:* Спецификация требует smoke-тесты, но не перечисляет конкретные случаи; конституция требует минимум 60% покрытия и smoke-тесты рендеринга ключевых компонентов.

**Options:**
- A) Только тест рендеринга без ошибок (компонент монтируется) + тест вызова onStartGame для каждого из трёх сценариев (PvP, PvAI+X, PvAI+O)
- B) Только тест рендеринга без ошибок (минимальный smoke)
- C) Полный набор: рендеринг + все три сценария вызова onStartGame + проверка видимости блока выбора стороны

**Answer:**
Только тест рендеринга без ошибок (компонент монтируется) + тест вызова onStartGame для каждого из трёх сценариев (PvP, PvAI+X, PvAI+O)
