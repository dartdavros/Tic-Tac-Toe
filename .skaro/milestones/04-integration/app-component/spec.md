# Specification: app-component

## Context
Компонент `App` является корневым компонентом приложения. Он владеет единственным `useReducer` с `gameReducer` и `INITIAL_STATE`, управляет условным рендерингом трёх экранов и передаёт `dispatch` в дочерние компоненты через callback-функции. Согласно архитектурным инвариантам, все мутации состояния происходят только через `dispatch`.

Переходы между экранами атомарны: `gameReducer` устанавливает `screen` в нужное значение в рамках одного действия, промежуточных состояний не существует. В частности, при завершении партии редьюсер немедленно устанавливает `screen === 'result'` — отдельного «переходного» состояния между `'game'` и `'result'` нет.

## User Scenarios
1. **Первый запуск:** пользователь видит экран меню (`GameSettings`).
2. **Начало игры:** пользователь настраивает игру и нажимает «Начать» — отображается игровое поле (`Board` + `StatusBar`).
3. **Завершение партии:** после победы или ничьей `gameReducer` атомарно переключает `screen` на `'result'`, автоматически отображается `ResultScreen`.
4. **«Играть снова»:** пользователь нажимает кнопку — возврат в меню.
5. **«В меню»:** пользователь нажимает кнопку — возврат в меню.
6. **Ошибка рендеринга:** `ErrorBoundary` перехватывает ошибку и показывает `ErrorFallback`. Пользователь нажимает кнопку сброса — состояние возвращается в `INITIAL_STATE`, `ErrorBoundary` сбрасывается.

## Functional Requirements
- FR-01: `App` использует `useReducer(gameReducer, INITIAL_STATE)` как единственный источник состояния.
- FR-02: При `state.screen === 'menu'` рендерится `<GameSettings onStartGame={...} />`.
- FR-03: При `state.screen === 'game'` рендерится `<Board ... />` и `<StatusBar ... />`.
- FR-04: При `state.screen === 'result'` рендерится `<ResultScreen ... />`.
- FR-05: `onStartGame` диспатчит `{ type: 'START_GAME', payload: settings }`.
- FR-06: `onCellClick` диспатчит `{ type: 'MAKE_MOVE', payload: { index } }`.
- FR-07: `onRestart` диспатчит `{ type: 'RESTART' }`.
- FR-08: `onQuitToMenu` диспатчит `{ type: 'QUIT_TO_MENU' }`.
- FR-09: Всё дерево компонентов ниже `App` обёрнуто в `<ErrorBoundary fallback={<ErrorFallback onReset={handleReset} />}>`.
- FR-10: `handleReset` выполняет два действия в следующем порядке: (1) диспатчит `{ type: 'RESTART' }`, что полностью сбрасывает состояние до `INITIAL_STATE` (экран меню, пустая доска); (2) инкрементирует локальный счётчик сбросов `resetKey` (хранится в `useState`), который передаётся как `key`-проп на `<ErrorBoundary>`, принудительно размонтируя и пересоздавая `ErrorBoundary`.
- FR-11: `Board` получает `disabled={state.status.kind !== 'playing'}` для блокировки ходов при завершённой партии.
- FR-12: `Board` получает `winLine` из `state.status` при `status.kind === 'won'`, в остальных случаях передаётся `undefined`.
- FR-13: Создан `src/components/App.test.tsx` со smoke-тестом начального рендера (экран меню отображается без ошибок).
- FR-14: Стили `App` определены в `src/styles/App.module.css` (центрирование, максимальная ширина контейнера).
- FR-15: `Board` получает ровно четыре пропа: `board`, `onCellClick`, `disabled`, `winLine`. Пропы `currentPlayer` и `settings` в `Board` не передаются — они передаются в `StatusBar`.
- FR-16: `main.tsx` оборачивает `<App />` в `<React.StrictMode>`. Двойной вызов редьюсера и эффектов в dev-режиме принимается как штатное поведение. `gameReducer` обязан быть чистой функцией без сайд-эффектов, чтобы двойной вызов не приводил к некорректному состоянию.

## Non-Functional Requirements
- NFR-01: Первый рендер приложения происходит без ошибок в консоли.
- NFR-02: Переход между экранами не вызывает мигания или промежуточных состояний.
- NFR-03: `gameReducer` является чистой функцией — не производит сайд-эффектов, детерминирован при одинаковых входных данных. Это обеспечивает корректную работу в `React.StrictMode` при двойном вызове редьюсера в dev-режиме.

## Boundaries (что НЕ входит)
- Не реализуется роутинг (URL не меняется при смене экрана).
- Не реализуется анимация переходов между экранами.
- Не реализуется персистентность состояния между сессиями.
- Не существует промежуточного состояния между `screen === 'game'` и `screen === 'result'`: переход атомарен внутри редьюсера.

## Acceptance Criteria
- [ ] При запуске приложения отображается экран меню.
- [ ] После нажатия «Начать игру» отображается игровое поле.
- [ ] После завершения партии отображается экран результата без промежуточного состояния.
- [ ] «Играть снова» и «В меню» возвращают в меню.
- [ ] `ErrorBoundary` оборачивает всё дерево компонентов ниже `App`.
- [ ] После срабатывания `ErrorBoundary` нажатие кнопки сброса: (а) возвращает состояние в `INITIAL_STATE`; (б) сбрасывает сам `ErrorBoundary` через инкремент `resetKey`; (в) отображает экран меню.
- [ ] `Board` получает ровно четыре пропа: `board`, `onCellClick`, `disabled`, `winLine` — не больше и не меньше.
- [ ] `main.tsx` содержит `<React.StrictMode>` вокруг `<App />`.
- [ ] `gameReducer` при двойном вызове с одинаковыми аргументами возвращает эквивалентный результат (чистота функции).
- [ ] Smoke-тест в `App.test.tsx` проходит (начальный рендер, экран меню виден).
- [ ] `npm run lint` не выдаёт ошибок.
- [ ] `npm run build` завершается без ошибок.
