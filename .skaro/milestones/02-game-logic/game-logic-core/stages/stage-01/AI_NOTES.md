## Почему такой подход

### Сигнатура `checkWinner`

Согласно уточнению Q1, функция проверяет победителя среди **обоих** игроков (X и O),
а параметр `currentPlayer` используется исключительно для формирования статуса `{ kind: 'playing', currentPlayer }`.
Это соответствует семантике: функция отвечает на вопрос «что происходит на доске сейчас?»,
а не «выиграл ли конкретный игрок?».

### Приоритет статусов (Q3)

Победа проверяется до заполненности доски — цикл по `WIN_LINES` выполняется первым.
Это корректно обрабатывает граничный случай: последний ход одновременно заполняет доску
и создаёт выигрышную линию.

### `applyMove` — порядок проверок

Проверка диапазона индекса (`index < 0 || index > 8`) выполняется **до** проверки занятости клетки.
Это логично: обращение к `board[-1]` в JavaScript не выбросит ошибку (вернёт `undefined`),
но семантически это программная ошибка, которую нужно поймать первой.

### Сохранение `computeGameStatus` и `getOpponent`

Эти функции уже использовались в существующих файлах `aiPlayer.ts` и `gameReducer.ts`
(судя по их импортам). Удаление нарушило бы компиляцию. Функции оставлены и экспортированы.

- `computeGameStatus(board, playerWhoJustMoved)` — удобная обёртка: автоматически
  вычисляет следующего игрока через `getNextPlayer` и вызывает `checkWinner`.
- `getOpponent` — псевдоним для `getNextPlayer`, используется в `aiPlayer.ts`
  для ясности намерений (семантика «противник» vs «следующий игрок»).

### `WIN_LINES` — тип

Использован `readonly [number, number, number][]` вместо `readonly number[][]`
для более строгой типизации (каждая линия — кортеж ровно из 3 элементов).
Это соответствует инварианту №12 (строгая типизация).

### Тесты

- Каждая из 8 выигрышных линий покрыта отдельным тестом (4 для X, 4 для O).
- Тест на приоритет победы над ничьёй использует реальную доску, где X побеждает
  по диагонали [0,4,8] при полностью заполненной доске.
- Тест на `winner` ≠ `currentPlayer` явно передаёт `currentPlayer: 'X'` при победе O.
- Тесты `applyMove` проверяют точные тексты сообщений об ошибках (AC из спецификации).
- Тест на порядок возрастания в `getAvailableMoves` проверяет инвариант попарно.

## Риски и ограничения

- **Обратная совместимость:** `aiPlayer.ts` импортирует `checkWinner` с сигнатурой
  `checkWinner(board)` (без `currentPlayer`), а `gameReducer.ts` использует `computeGameStatus`
  и `getOpponent`. Новая сигнатура `checkWinner(board, currentPlayer)` **требует обязательного**
  второго аргумента — это может сломать существующий `aiPlayer.ts`, который вызывает
  `checkWinner(board)` без второго аргумента. Необходимо проверить и обновить `aiPlayer.ts`.

- **Альтернатива для `checkWinner`:** можно было сделать `currentPlayer` опциональным
  (`currentPlayer?: Player`) и возвращать `{ kind: 'playing', currentPlayer: currentPlayer ?? 'X' }`
  по умолчанию. Это сохранило бы обратную совместимость, но нарушило бы строгость типов
  (спецификация требует явного параметра). Выбрана строгая сигнатура согласно FR-01.
