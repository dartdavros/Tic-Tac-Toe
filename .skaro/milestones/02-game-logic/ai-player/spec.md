# Specification: ai-player

## Context
Алгоритм ИИ обеспечивает режим игры против компьютера (PvAI). Согласно ADR-002, используется minimax с полным перебором без оптимизаций — для поля 3×3 это тривиально по производительности (< 5 мс). Алгоритм должен быть детерминированным для тестируемости. Согласно архитектурному инварианту №4, `getBestMove` вызывается только из `gameReducer`.

## User Scenarios
1. **ИИ может выиграть за один ход:** `getBestMove` возвращает индекс победного хода.
2. **Противник может выиграть за один ход:** `getBestMove` возвращает индекс блокирующего хода.
3. **Пустая доска (первый ход):** `getBestMove` возвращает детерминированный оптимальный ход (например, центр — индекс 4).
4. **Полная доска:** `getBestMove` выбрасывает `Error('No available moves')`.

## Functional Requirements
- FR-01: Реализована функция `getBestMove(board: BoardState, aiPlayer: Player): number`. Возвращает индекс оптимального хода [0..8].
- FR-02: Алгоритм — minimax с полным перебором. ИИ максимизирует свой счёт и минимизирует счёт противника.
- FR-03: Функция детерминирована: при одинаковом `board` и `aiPlayer` всегда возвращает одинаковый индекс.
- FR-04: При полностью заполненной доске (`isBoardFull` возвращает `true`) функция выбрасывает `Error('No available moves')`.
- FR-05: Реализована вспомогательная функция `minimax(board: BoardState, aiPlayer: Player, currentPlayer: Player, isMaximizing: boolean): number`. Возвращает оценку позиции: +10 (победа ИИ), -10 (победа противника), 0 (ничья).
- FR-06: Функция `getBestMove` экспортируется из `src/logic/aiPlayer.ts`.
- FR-07: Длина каждой функции не превышает 40 строк.

## Non-Functional Requirements
- NFR-01: `getBestMove` выполняется менее чем за 50 мс на любом валидном состоянии доски (проверяется автоматическим тестом).
- NFR-02: Покрытие тестами модуля `aiPlayer.ts` ≥ 90%.
- NFR-03: Функция не использует глобальное состояние и не имеет побочных эффектов.

## Boundaries (что НЕ входит)
- Не реализуется alpha-beta pruning (согласно ADR-002, избыточно для поля 3×3).
- Не реализуется случайный выбор среди равнозначных ходов (алгоритм детерминирован).
- Не реализуется Web Worker (синхронный вызов достаточен для поля 3×3).

## Acceptance Criteria
- [ ] ИИ всегда выбирает победный ход, если он доступен.
- [ ] ИИ всегда блокирует победный ход противника, если нет собственного победного хода.
- [ ] На пустой доске `getBestMove` возвращает один и тот же индекс при повторных вызовах (детерминированность).
- [ ] `getBestMove` на полной доске выбрасывает `Error('No available moves')`.
- [ ] Тест производительности: `getBestMove(EMPTY_BOARD, 'X')` выполняется менее чем за 50 мс.
- [ ] Покрытие тестами ≥ 90%.
- [ ] `npm run lint` не выдаёт ошибок.

## Open Questions
- Нужно ли добавить небольшую случайность при равнозначных ходах для более «человечной» игры, или строгий детерминизм важнее для тестируемости?
- Следует ли оптимизировать оценку с учётом глубины дерева (предпочитать более быструю победу)?
