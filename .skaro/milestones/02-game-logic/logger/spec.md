# Specification: logger

## Context
Логгер используется редьюсером (`reducer_error`) и `ErrorBoundary` (`render_error`) для записи событий. Согласно архитектуре, в dev-режиме события пишутся в консоль, в prod — модуль является точкой расширения для подключения аналитики (например, Vercel Analytics). Модуль должен быть реализован до редьюсера, так как редьюсер его импортирует.

## User Scenarios
1. **Ошибка в редьюсере (dev):** в консоли браузера появляется сообщение `[logEvent] reducer_error { action: ..., error: ... }`.
2. **Ошибка рендеринга (dev):** в консоли появляется `[logEvent] render_error { ... }`.
3. **Production-сборка:** вызов `logEvent` не вызывает ошибок; в будущем здесь можно подключить внешний сервис аналитики.

## Functional Requirements
- FR-01: Реализована функция `logEvent(event: LogEventType, payload?: unknown): void`, где `LogEventType` — литеральный union `'reducer_error' | 'render_error'`. Строгая типизация гарантирует ошибку компиляции при передаче неизвестного типа события.
- FR-02: В режиме `development` (`import.meta.env.DEV === true`) функция вызывает `console.warn('[logEvent]', event, payload ?? '')`. Используется исключительно `console.warn` — единообразно для всех типов событий.
- FR-03: В режиме `production` функция является заглушкой (no-op) с комментарием `// TODO: подключить аналитику в production`.
- FR-04: Функция и тип `LogEventType` экспортируются из `src/utils/logger.ts`.
- FR-05: Функция не выбрасывает исключений ни при каких входных данных. Тело функции оборачивается в `try/catch`: при перехвате исключения внутри логгера ошибка молча игнорируется, чтобы логгер никогда не нарушал работу вызывающего кода.

## Non-Functional Requirements
- NFR-01: Вызов `logEvent` в production не увеличивает бандл более чем на 100 байт.
- NFR-02: Функция не блокирует основной поток (синхронная, без сетевых запросов на данном этапе).

## Тестирование
- Отдельные юнит-тесты для `logEvent` не пишутся.
- Покрытие обеспечивается косвенно через тесты редьюсера, в которых `logEvent` вызывается при обработке ошибок.
- В тестах редьюсера, где требуется подавить вывод в консоль, используется `vi.stubEnv('DEV', false)` для явного выставления `import.meta.env.DEV` в `false`. Это гарантирует, что `console.warn` не вызывается в тестовой среде и вывод остаётся чистым.

## Boundaries (что НЕ входит)
- Не реализуется отправка событий на внешний сервер (только точка расширения).
- Не реализуется очередь событий или батчинг.
- Не реализуются уровни логирования (info, warn, error) — только единственная функция `logEvent`.

## Acceptance Criteria
- [ ] `logEvent('reducer_error', { foo: 'bar' })` в dev-режиме выводит в консоль сообщение через `console.warn` с префиксом `[logEvent]`.
- [ ] `logEvent('render_error')` в dev-режиме выводит в консоль сообщение через `console.warn`; `payload` при этом отображается как пустая строка `''`.
- [ ] `logEvent` не выбрасывает исключений при `payload === undefined`.
- [ ] `logEvent` не выбрасывает исключений при `payload === null`.
- [ ] `logEvent` не выбрасывает исключений даже если внутри `try`-блока возникает непредвиденная ошибка (гарантируется `try/catch`).
- [ ] TypeScript выдаёт ошибку компиляции при вызове `logEvent('unknown_event')` — тип `event` ограничен union `'reducer_error' | 'render_error'`.
- [ ] В production-режиме (`import.meta.env.DEV === false`) вызов `logEvent` не производит никаких побочных эффектов.
- [ ] Файл `src/utils/logger.ts` создан, экспортирует функцию `logEvent` и тип `LogEventType`.
- [ ] `npm run lint` не выдаёт ошибок.
