# Specification: logger

## Context
Логгер используется редьюсером (`reducer_error`) и `ErrorBoundary` (`render_error`) для записи событий. Согласно архитектуре, в dev-режиме события пишутся в консоль, в prod — модуль является точкой расширения для подключения аналитики (например, Vercel Analytics). Модуль должен быть реализован до редьюсера, так как редьюсер его импортирует.

## User Scenarios
1. **Ошибка в редьюсере (dev):** в консоли браузера появляется сообщение `[logEvent] reducer_error { action: ..., error: ... }`.
2. **Ошибка рендеринга (dev):** в консоли появляется `[logEvent] render_error { ... }`.
3. **Production-сборка:** вызов `logEvent` не вызывает ошибок; в будущем здесь можно подключить внешний сервис аналитики.

## Functional Requirements
- FR-01: Реализована функция `logEvent(event: string, payload?: unknown): void`.
- FR-02: В режиме `development` (`import.meta.env.DEV === true`) функция вызывает `console.warn('[logEvent]', event, payload ?? '')`.
- FR-03: В режиме `production` функция является заглушкой (no-op) или вызывает внешний сервис аналитики — на данном этапе реализуется как no-op с комментарием `// TODO: подключить аналитику в production`.
- FR-04: Функция экспортируется из `src/utils/logger.ts`.
- FR-05: Функция не выбрасывает исключений ни при каких входных данных.

## Non-Functional Requirements
- NFR-01: Вызов `logEvent` в production не увеличивает бандл более чем на 100 байт.
- NFR-02: Функция не блокирует основной поток (синхронная, без сетевых запросов на данном этапе).

## Boundaries (что НЕ входит)
- Не реализуется отправка событий на внешний сервер (только точка расширения).
- Не реализуется очередь событий или батчинг.
- Не реализуются уровни логирования (info, warn, error) — только единственная функция `logEvent`.

## Acceptance Criteria
- [ ] `logEvent('test', { foo: 'bar' })` в dev-режиме выводит сообщение в консоль.
- [ ] `logEvent` не выбрасывает исключений при `payload === undefined`.
- [ ] `logEvent` не выбрасывает исключений при `payload === null`.
- [ ] Файл `src/utils/logger.ts` создан и экспортирует функцию `logEvent`.
- [ ] `npm run lint` не выдаёт ошибок.

## Open Questions
- Нужно ли типизировать `event` как литеральный union (`'reducer_error' | 'render_error' | ...`) для строгости, или `string` достаточно?
- Стоит ли добавить `logEvent` в тесты редьюсера через мок, чтобы проверить, что он вызывается при ошибках?
