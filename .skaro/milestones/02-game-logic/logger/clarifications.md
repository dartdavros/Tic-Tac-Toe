# Clarifications: logger

## Question 1
Нужно ли типизировать параметр `event` как литеральный union (`'reducer_error' | 'render_error'`) вместо `string`?

*Context:* Строгая типизация предотвращает опечатки при вызове logEvent в редьюсере и ErrorBoundary, но требует обновления типа при добавлении новых событий.

**Options:**
- A) Использовать литеральный union: `'reducer_error' | 'render_error'` — строгая типизация, ошибки на этапе компиляции
- B) Использовать `string` — гибко, не требует обновления типа при расширении
- C) Использовать `string` с экспортируемым вспомогательным типом `LogEvent = 'reducer_error' | 'render_error'` для опциональной строгости

**Answer:**
Использовать литеральный union: `'reducer_error' | 'render_error'` — строгая типизация, ошибки на этапе компиляции

## Question 2
Должен ли `logEvent` в dev-режиме использовать `console.warn` или `console.log`?

*Context:* Спецификация (FR-02) явно указывает `console.warn`, но для информационных событий (не ошибок) `warn` создаёт визуальный шум в консоли; важно зафиксировать намерение.

**Options:**
- A) Всегда `console.warn` — как указано в FR-02, единообразно для всех событий
- B) Использовать `console.error` для событий с суффиксом `_error`, `console.warn` для остальных
- C) Использовать `console.log` — менее навязчиво, достаточно для dev-режима

**Answer:**
Всегда `console.warn` — как указано в FR-02, единообразно для всех событий

## Question 3
Нужно ли оборачивать тело `logEvent` в `try/catch` для гарантии FR-05 (функция не выбрасывает исключений)?

*Context:* Вызов `console.warn` теоретически безопасен, но сериализация `payload` (например, циклические ссылки) может вызвать исключение в некоторых окружениях.

**Options:**
- A) Добавить `try/catch` внутри `logEvent` — полная гарантия FR-05 при любых входных данных
- B) Не добавлять `try/catch` — `console.warn` достаточно надёжен, усложнение не оправдано
- C) Добавить `try/catch` только вокруг вызова `console.warn`, игнорируя ошибку в `catch`

**Answer:**
Добавить `try/catch` внутри `logEvent` — полная гарантия FR-05 при любых входных данных

## Question 4
Нужно ли покрывать `logEvent` юнит-тестами, и если да — как мокировать `console.warn` и `import.meta.env.DEV`?

*Context:* ADR-008 требует co-location тестов; спецификация упоминает тестирование как открытый вопрос, но минимальное покрытие 60% подразумевает наличие тестов для утилит.

**Options:**
- A) Написать тесты для `logEvent`: мокировать `console.warn` через `vi.spyOn`, `import.meta.env.DEV` через `vi.stubEnv` в Vitest
- B) Не писать отдельные тесты для логгера — покрытие обеспечивается тестами редьюсера, где logEvent вызывается косвенно
- C) Написать только smoke-тест: вызов `logEvent` не выбрасывает исключений (проверка FR-05)

**Answer:**
Не писать отдельные тесты для логгера — покрытие обеспечивается тестами редьюсера, где logEvent вызывается косвенно

## Question 5
Как определить значение `import.meta.env.DEV` в тестовой среде Vitest?

*Context:* В тестах Vitest `import.meta.env.DEV` может быть `true` по умолчанию, что означает, что `console.warn` будет вызываться во всех тестах, включая тесты редьюсера, создавая шум в выводе.

**Options:**
- A) Явно мокировать `import.meta.env.DEV = false` в тестах редьюсера через `vi.stubEnv('DEV', false)` чтобы подавить вывод
- B) Оставить как есть — шум в консоли при тестах допустим для учебного проекта
- C) Добавить в `vitest.config.ts` глобальный `env: { DEV: false }` для всех тестов

**Answer:**
Явно мокировать `import.meta.env.DEV = false` в тестах редьюсера через `vi.stubEnv('DEV', false)` чтобы подавить вывод
