# Clarifications: typescript-types

## Question 1
Нужен ли вспомогательный тип `WinLine = number[]` или достаточно inline-определения `winLine: number[]` внутри ветки `won` типа `GameStatus`?

*Context:* Если `WinLine` будет использоваться в нескольких местах (логика победителя, подсветка клеток), отдельный тип улучшит читаемость и рефакторинг; если только в одном месте — избыточен.

**Options:**
- A) Оставить inline: `winLine: number[]` прямо в ветке `won` — отдельный тип не нужен
- B) Добавить `export type WinLine = number[]` и использовать его в `GameStatus` и других модулях
- C) Добавить `export type WinLine = [number, number, number]` как кортеж фиксированной длины 3 (все выигрышные линии ровно из 3 клеток)

**Answer:**
Оставить inline: `winLine: number[]` прямо в ветке `won` — отдельный тип не нужен

## Question 2
Нужен ли тип `CellIndex = 0 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8` для строгой типизации индексов клеток?

*Context:* Тип `CellIndex` позволит TypeScript отловить выход за пределы доски на этапе компиляции, но потребует явных приведений типов (`as CellIndex`) в логике minimax и редьюсере, что увеличит шум в коде.

**Options:**
- A) Не нужен — использовать `number` для индексов клеток везде, валидация на уровне runtime в gameReducer
- B) Добавить `export type CellIndex = 0 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8` и использовать в `BoardState` и `GameAction.MAKE_MOVE`
- C) Добавить тип, но использовать только в `GameAction` (`payload: { index: CellIndex }`), не меняя сигнатуру `BoardState`

**Answer:**
Не нужен — использовать `number` для индексов клеток везде, валидация на уровне runtime в gameReducer

## Question 3
Должен ли `GameState` включать поле для хранения истории ходов (например, `history: BoardState[]`) с учётом возможной функции «отмены хода»?

*Context:* Спецификация не упоминает отмену ходов, но ADR-001 фиксирует единственный объект `GameState` — добавление истории сейчас или позже потребует изменения контракта типов и редьюсера.

**Options:**
- A) Не включать — история ходов вне scope текущей спецификации, `GameState` без `history`
- B) Добавить `history: BoardState[]` в `GameState` уже сейчас как пустой массив, чтобы не ломать контракт позже
- C) Добавить опциональное поле `history?: BoardState[]` для совместимости без обязательного использования

**Answer:**
Не включать — история ходов вне scope текущей спецификации, `GameState` без `history`

## Question 4
Как типизировать `GameSettings` в начальном состоянии `GameState` до того, как пользователь выбрал режим игры (экран `menu`)?

*Context:* На экране `menu` настройки ещё не выбраны, но `GameState.settings: GameSettings` обязателен согласно FR-08 — нужно либо допустить `null`, либо задать дефолтное значение, либо сделать поле опциональным.

**Options:**
- A) Сделать поле опциональным: `settings?: GameSettings` — на экране menu поле отсутствует
- B) Добавить третью ветку в union: `{ mode: 'none' }` как начальное/неинициализированное состояние
- C) Хранить дефолтное значение `{ mode: 'pvp' }` как начальное состояние — поле всегда определено
- D) Изменить тип на `GameSettings | null` и обрабатывать `null` в редьюсере и компонентах

**Answer:**
Сделать поле опциональным: `settings?: GameSettings` — на экране menu поле отсутствует

## Question 5
Должен ли тип `GameAction` включать действие `RESET_ERROR` или аналогичное для сброса состояния ошибки, учитывая ADR-007 (защитный gameReducer с try/catch)?

*Context:* ADR-007 описывает возврат текущего состояния при ошибке, но не определяет, как пользователь может выйти из «застрявшего» состояния — нужно ли отдельное действие или достаточно `QUIT_TO_MENU`.

**Options:**
- A) Не нужен — `QUIT_TO_MENU` достаточно для сброса любого некорректного состояния
- B) Добавить `{ type: 'RESET_ERROR' }` в `GameAction` для явного сброса ошибочного состояния к начальному
- C) Обработка ошибок полностью на уровне `ErrorBoundary` (ADR-005), в `GameAction` ничего добавлять не нужно

**Answer:**
Не нужен — `QUIT_TO_MENU` достаточно для сброса любого некорректного состояния
