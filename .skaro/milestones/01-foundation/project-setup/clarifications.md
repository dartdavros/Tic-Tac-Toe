# Clarifications: project-setup

## Question 1
Нужно ли настраивать алиасы путей (например, `@/` → `src/`) в `tsconfig.json` и `vite.config.ts`?

*Context:* Алиасы влияют на все последующие импорты в проекте — если добавить их позже, придётся переписывать пути во всех файлах.

**Options:**
- A) Да, настроить `@/` → `src/` в обоих файлах (`tsconfig.json` paths + `vite.config.ts` resolve.alias)
- B) Нет, использовать относительные пути (`../`, `./`) — алиасы не нужны для проекта такого размера
- C) Настроить только в `tsconfig.json` для проверки типов, без `vite.config.ts`

**Answer:**
Да, настроить `@/` → `src/` в обоих файлах (`tsconfig.json` paths + `vite.config.ts` resolve.alias)

## Question 2
Нужно ли подключать `eslint-plugin-jsx-a11y` для автоматической проверки доступности (a11y) на этапе настройки проекта?

*Context:* ADR-010 требует обязательного минимума a11y-атрибутов; плагин позволяет автоматически проверять их через линтер, но увеличивает сложность конфигурации.

**Options:**
- A) Да, установить и настроить `eslint-plugin-jsx-a11y` с рекомендуемыми правилами (`recommended`)
- B) Нет, a11y проверяется вручную и через тесты — плагин не нужен на данном этапе
- C) Установить плагин, но включить только критичные правила (не `recommended`), остальные — `warn`

**Answer:**
Нет, a11y проверяется вручную и через тесты — плагин не нужен на данном этапе

## Question 3
Что должен рендерить `main.tsx` после удаления стандартных заглушек Vite — пустой `<div>` или минимальный компонент-заглушку `<App />`?

*Context:* Спецификация требует 'пустую страницу без ошибок в консоли', но не уточняет, должен ли существовать компонент `App.tsx` на данном этапе.

**Options:**
- A) Рендерить `<div id='root' />` без каких-либо компонентов — `App.tsx` не создаётся
- B) Создать минимальный `App.tsx`, возвращающий `<></>` (пустой фрагмент), и рендерить его в `main.tsx`
- C) Рендерить строку-заглушку `<h1>Tic-Tac-Toe</h1>` без отдельного компонента `App`

**Answer:**
Рендерить `<div id='root' />` без каких-либо компонентов — `App.tsx` не создаётся

## Question 4
Нужно ли настраивать `husky` + `lint-staged` для pre-commit хуков на данном этапе?

*Context:* Pre-commit хуки обеспечивают автоматическую проверку кода перед каждым коммитом, но усложняют первоначальную настройку и могут мешать быстрым итерациям на старте.

**Options:**
- A) Да, настроить `husky` + `lint-staged` для запуска `lint` и `format:check` перед каждым коммитом
- B) Нет, не нужно — проект учебный, pre-commit хуки избыточны
- C) Добавить `husky`, но без `lint-staged` — только запуск `npm test` перед коммитом

**Answer:**
Нет, не нужно — проект учебный, pre-commit хуки избыточны

## Question 5
Какой минимальный порог покрытия тестами (`thresholds`) указать в `vitest.config.ts`, учитывая что на этапе настройки тестов ещё нет?

*Context:* Если порог выставлен в 60% (как в конституции), то `npm run test:coverage` будет падать с ошибкой на пустом проекте, блокируя CI.

**Options:**
- A) Выставить порог 60% согласно конституции — пусть `test:coverage` падает до появления реальных тестов
- B) Выставить порог 0% сейчас, поднять до 60% в задаче написания тестов
- C) Не указывать `thresholds` вовсе в `vitest.config.ts` — добавить позже отдельным коммитом

**Answer:**
Выставить порог 60% согласно конституции — пусть `test:coverage` падает до появления реальных тестов

## Question 6
Какую версию `@types/react` и `@types/react-dom` использовать, и нужно ли явно фиксировать версии зависимостей в `package.json`?

*Context:* Vite-шаблон `react-ts` может установить `@types/react@18` или `@types/react@19` в зависимости от версии шаблона, что влияет на совместимость типов.

**Options:**
- A) Явно зафиксировать `@types/react@^18` и `@types/react-dom@^18` в `package.json`
- B) Использовать версии, которые установит шаблон `npm create vite@latest` по умолчанию, без ручной фиксации
- C) Использовать `@types/react@^19` — последнюю стабильную версию типов

**Answer:**
Использовать версии, которые установит шаблон `npm create vite@latest` по умолчанию, без ручной фиксации
