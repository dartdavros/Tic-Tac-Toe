## Почему такой подход

### types/index.ts
Типы реализованы точно по архитектурной спецификации. `BoardState` — кортеж фиксированной длины 9 (инвариант №12). `GameSettings` — дискриминированный union (инвариант №13, ADR-007). `GameAction` не содержит `GAME_OVER` — переход на `result` происходит автоматически в редьюсере (инвариант №5).

### logger.ts
Использует `import.meta.env.DEV` для определения окружения (стандартный способ в Vite). В dev пишет через `console.warn` (ESLint разрешает `warn` и `error`). В prod — точка расширения для аналитики.

### gameLogic.ts
Все функции — чистые (нет побочных эффектов, нет импортов из React или компонентов). `applyMove` иммутабелен — создаёт копию через spread. `computeGameStatus` объединяет `checkWinner` + `isBoardFull` для удобства редьюсера.

### aiPlayer.ts
Реализован классический minimax без alpha-beta pruning (ADR-002). Алгоритм детерминирован: при одинаковом состоянии всегда возвращает первый найденный оптимальный ход (перебор в порядке индексов). Для поля 3×3 время выполнения << 50 мс (инвариант №7). При пустой доске выбирает центр (индекс 4) — это оптимальный первый ход по minimax.

### gameReducer.ts
- `MAKE_MOVE` выполняет полную валидацию перед применением хода (инварианты безопасности)
- Ход ИИ вычисляется синхронно внутри редьюсера (ADR-003, инвариант №4)
- Переход на `result` происходит только внутри редьюсера (инвариант №5)
- `try/catch` оборачивает всю логику (инвариант №6, ADR-008)
- `RESTART` и `QUIT_TO_MENU` оба возвращают `INITIAL_STATE` — это соответствует архитектуре (оба действия ведут в меню)

## Риски и ограничения

1. **`noUncheckedIndexedAccess`**: В `tsconfig.json` включён `noUncheckedIndexedAccess`. В `aiPlayer.ts` при обращении к `moves[0]` используется приведение `as number` после явной проверки `moves.length > 0`. Это безопасно, но требует внимания при рефакторинге.

2. **Детерминированность minimax**: Алгоритм выбирает первый найденный ход с максимальной оценкой. Тест `'выбирает центр на пустой доске'` предполагает, что центр (индекс 4) будет выбран первым среди оптимальных ходов. Это верно для текущей реализации (перебор в порядке возрастания индексов), но может измениться при оптимизации.

3. **Поведение ИИ когда человек играет за O**: В режиме PvAI, если `humanPlayer === 'O'`, то первый ход делает X (ИИ). Текущая реализация редьюсера корректно обрабатывает этот случай: после `START_GAME` статус `currentPlayer: 'X'`, и если X — это ИИ, то редьюсер ждёт хода человека (O). Однако первый ход ИИ не делается автоматически при старте игры — это потребует обработки в компоненте `App` через `useEffect`. Это известное ограничение данного этапа.

4. **Покрытие тестами**: Тесты написаны для критической логики. Общее покрытие проекта пока ниже 60% из-за отсутствия тестов компонентов — это ожидаемо на данном этапе.

## Как проверить

1. `npm run build` — должен завершиться без ошибок TypeScript
2. `npm run lint` — должен завершиться без ошибок ESLint
3. `npm test` — все тесты должны пройти (gameLogic, aiPlayer, gameReducer)
4. `npm run test:coverage` — покрытие модулей `logic/` должно быть ≥ 90% (общее может быть ниже 60% до реализации компонентов)
5. Вручную проверить детерминированность: запустить `getBestMove(EMPTY, 'X')` несколько раз — результат всегда 4 (центр)
